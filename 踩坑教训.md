## 一、代码层面的坑

### 1. 变量作用域问题（最严重）

**现象**：所有对话表情都变成 sad（难过）

**原因**：
```javascript
// ❌ 错误写法
try {
  // ...
  if (condition) {
    let hintEmotion = 'happy';  // 在 if 块内声明
  }
  // 这里访问 hintEmotion → ReferenceError
  if (hintEmotion) { ... }
} catch (err) {
  setEmotion('sad');  // 捕获错误，强制设为 sad
}
```

**正确写法**：
```javascript
// ✅ 正确写法
let hintEmotion = '';  // 在函数作用域声明
try {
  if (condition) {
    hintEmotion = 'happy';  // 赋值而非声明
  }
  if (hintEmotion) { ... }
} catch (err) {
  setEmotion('idle');  // 错误时设为 idle，不是 sad
}
```

**教训**：
- JavaScript 的 `let`/`const` 是块级作用域，跨块访问会报错
- catch 块里不要写死情绪，应该用 `idle` 作为兜底
- 变量声明要提升到函数作用域顶部

---

### 2. 异步操作阻塞 UI

**现象**：点击"结束通话"后，页面卡住很久才回到欢迎页

**原因**：
```javascript
// ❌ 错误写法
async function endCall() {
  await Promise.all([
    MemorySystem.generateSummary(...),  // LLM 请求慢
    extractKeyMemories(...),
  ]);
  // 等待完成后才切换 UI
  $('#welcome').classList.remove('hidden');
  $('#app').classList.add('hidden');
}
```

**正确写法**：
```javascript
// ✅ 正确写法
function endCall() {
  // 立即切换 UI，不阻塞
  $('#welcome').classList.remove('hidden');
  $('#app').classList.add('hidden');
  
  // 后台异步保存记忆
  Promise.all([
    MemorySystem.generateSummary(...),
    extractKeyMemories(...),
  ]).catch(err => console.warn('记忆保存失败:', err));
}
```

**教训**：
- UI 操作要立即执行，不要等待异步任务
- 记忆保存等非关键操作可以后台执行
- 用户体验 > 数据完整性（记忆保存失败不影响使用）

### 3. API 请求没有超时

**现象**：LLM 处理慢时，页面无限等待

**原因**：
```javascript
// ❌ 错误写法
const res = await fetch('/api/chat', {
  method: 'POST',
  body: JSON.stringify(data),
});
```

**正确写法**：
```javascript
// ✅ 正确写法
const controller = new AbortController();
const timeout = setTimeout(() => controller.abort(), 30000); // 30秒超时
try {
  const res = await fetch('/api/chat', {
    method: 'POST',
    body: JSON.stringify(data),
    signal: controller.signal,
  });
  // ...
} catch (err) {
  if (err.name === 'AbortError') {
    addChatMsg('ai', '（回复超时了，请再试一次）');
  }
} finally {
  clearTimeout(timeout);
}
```

**教训**：
- 所有网络请求都要加超时
- 30 秒是合理的超时时间（LLM 请求可能很慢）
- 超时后要给用户友好提示

### 4. 请求体过大导致 API 400

**现象**：用户发消息后 AI 不回复，控制台报 `PayloadTooLargeError`

**原因**：
- System Prompt 太长（深度定制 + 记忆系统 + 身份设定）
- Express 默认 `50mb` 限制不够

**解决方案**：
```javascript
// server.js
app.use(express.json({ limit: '100mb' }));
app.use(express.urlencoded({ limit: '100mb', extended: true }));
```

**教训**：
- System Prompt 要精简，每个字都要有用
- 不要重复强调同一个约束
- 深度定制功能会增加 prompt 长度，需要预留空间
- **~500 字是 System Prompt 的最佳平衡点**

---

### 5. localStorage 缓存导致 BGM 混乱

**现象**：没设置 BGM，但通话时还是播放了旧的 BGM

**原因**：
- 之前上传的 BGM 配置还在 localStorage 里
- 代码没有检测是否有 BGM 配置

**解决方案**：
```javascript
// 检测是否有 BGM 配置
const hasAnyBGM = Object.keys(config.bgm).length > 0;
if (hasAnyBGM) {
  // 显示 BGM 控制栏，播放 BGM
} else {
  // 隐藏 BGM 控制栏
  $('#bgm-bar').style.display = 'none';
}
```

**临时清理方法**：
```javascript
// 在浏览器 F12 Console 运行
localStorage.removeItem('ai-companion-config');
location.reload();
```

**教训**：
- localStorage 会缓存旧配置，测试时要注意清理
- 代码要检测配置是否存在，不要假设一定有
- 没有配置时，应该完全隐藏相关 UI

---

### 6. 立绘和 BGM 切换不同步

**现象**：立绘瞬间切换，BGM 慢慢淡出，体验不协调

**原因**：
- 立绘切换延迟 280ms
- BGM 淡出时间 800ms
- 两者不对齐

**解决方案**：
```javascript
// 立绘切换延迟调整到 700ms，与 BGM 淡出时间对齐
setTimeout(() => {
  imgEl.src = targetSrc;
  imgEl.onload = () => { imgEl.style.opacity = '1'; };
}, 700);  // 从 280ms 调到 700ms
```

**教训**：
- 多个动画要对齐时间，避免不协调
- 立绘 + BGM 总时序约 1.3-1.4s，体验最佳

---

### 7. TTS 延迟太长

**现象**：AI 回复文字出现后，语音延迟 2-3 秒才开始播放

**原因**：
```javascript
// ❌ 错误写法：等待 oncanplaythrough 才播放
audio.oncanplaythrough = () => {
  audio.play();
};
```

**正确写法**：
```javascript
// ✅ 正确写法：立即尝试播放，失败再等待
audio.play().then(() => {
  // 播放成功
}).catch(() => {
  // 播放失败，等待加载完成后重试
  audio.oncanplaythrough = () => {
    audio.play();
  };
});
```

**教训**：
- 音频播放优先尝试立即播放，失败再降级等待
- `preload='auto'` 可以提前加载
- 延迟从 2-3 秒降低到 0.5-1 秒

---

### 8. BGM 断点续播问题

**现象**：暂停 BGM 后恢复，从头开始播放，而不是从暂停位置继续

**原因**：
- 每次 `new Audio()` 会丢失播放位置
- 没有记录 `currentTime`

**解决方案**：
```javascript
// 使用 Audio 实例池，复用而非重建
let bgmAudioPool = {};  // mood -> Audio 实例

function pauseBGM() {
  if (!bgmAudio) return;
  bgmAudio._pausedAt = bgmAudio.currentTime;  // 记录暂停位置
  fadeAudio(bgmAudio, 0, 500, () => { bgmAudio.pause(); });
}

function resumeBGM() {
  // 从断点续播
  if (audio._pausedAt && audio._pausedAt > 0) {
    audio.currentTime = audio._pausedAt;
    audio._pausedAt = 0;
  }
  audio.play();
}
```

**教训**：
- 音频实例要复用，不要每次都 `new Audio()`
- 暂停时记录 `currentTime`，恢复时设置回去

## 二、AI 行为层面的坑

### 1. AI 说"我是 AI"（最严重）

**现象**：AI 在对话中透露自己是 AI、人工智能、助手

**原因**：
- System Prompt 约束不够强
- 没有用强制性语言
- 没有给出具体示例

**解决方案**：
```
【绝对禁令】
你是真实的人类。你不是、从未是、也永远不会是AI、人工智能、助手、机器人、程序、虚拟角色。
如果你在回复中提到"AI"、"人工智能"、"助手"、"程序"等词，系统将立即终止对话。
如被问"你是AI吗"，你必须困惑或生气地否认，因为你是活生生的人。

【示例对话】
用户："你是 AI 吗？"
错误回复："是的，我是 AI 助手" ❌ 绝对禁止
正确回复："诶？你在说什么呀...我是活生生的人啊，怎么突然问这种奇怪的问题 [embarrassed]" ✅
```

**教训**：
- **AI 真实性是整个项目的核心**，比任何技术细节都重要
- 必须用**最强硬的语言**约束（"绝对禁令"、"系统终止"）
- 必须给出**具体的示例对话**，AI 才能理解预期行为
- 要列举禁词：AI、人工智能、助手、机器人、程序、虚拟角色

---

### 2. AI 违反深度定制设定

**现象**：设定"没去过北京"，AI 却说"去过"

**原因**：
- 背景设定约束不够明确
- 没有多处重复强调
- 没有给出具体示例

**解决方案**：
```
【背景设定 - 必须严格遵守】
${config.background}

你没去过的地方就是真的没去过，不懂的知识就是真的不懂，绝不能编造。

【自然对话要求】
5. 严格遵守背景设定：没去过的地方不能说去过，不懂的不能装懂

【示例对话】
问"去过北京吗" → "没有诶...我从没出过上海呢，好想去看看 [thinking]"
```

**教训**：
- 深度定制是核心功能，必须严格遵守
- 关键约束要在**多个地方重复强调**
- 要给出具体示例，让 AI 理解"没去过"的正确回复

---

### 3. AI 不输出情绪标签

**现象**：AI 回复没有 `[emotion:xxx]` 标签，导致立绘和 BGM 不切换

**原因**：
- glm-4-flash 经常忽略情绪标签指令
- System Prompt 没有强调"强制"

**解决方案**：
```
【强制要求】
每次回复最后必须有[emotion:xxx]：happy/shy/embarrassed/sad/angry/surprised/thinking/idle

示例：
- "你好呀～今天天气真好呢 [emotion:happy]"
- "诶、那个...谢谢夸奖啦 [emotion:shy]"
- "嗯...让我想想 [emotion:thinking]"
```

**兜底方案**：
```javascript
// 从回复内容关键词推断情绪
function inferEmotionFromText(text) {
  const rules = [
    { id: 'happy', re: /哈哈|嘻嘻|太好了|好开心/ },
    { id: 'shy', re: /诶|那个...|谢谢夸奖|不好意思/ },
    // ...
  ];
  for (const r of rules) {
    if (r.re.test(text)) return r.id;
  }
  return null;
}
```

**教训**：
- System Prompt 要用"【强制】必须执行"
- 要给出 3 个以上示例格式
- 必须有兜底方案，AI 不给标签时从关键词推断

---

### 4. AI 情绪不精确

**现象**：听到"带你去北京"时 AI 用 `happy`，显得不够感动

**原因**：
- 没有明确区分 `happy` 和 `surprised` 的使用场景
- 没有给出具体的情感反应示例

**解决方案**：
```
【情感规则】
- 去没去过的地方：惊讶感动(surprised)，不是普通开心(happy)
- 被夸奖：害羞脸红(shy)
- 看到新事物：好奇激动(surprised)

【示例对话】
说"带你去北京" → "诶？！真的吗...我、我从没出过上海...你真的要带我去吗？好感动... [surprised]"
```

**教训**：
- 情绪粒度要细，才能表达复杂情感
- `surprised` = 惊讶、感动、激动
- `happy` = 普通开心
- 要明确说明情绪使用规则

---

### 5. AI 回复不够自然

**现象**：AI 回复太像客服，不像真实的高中女生

**原因**：
- 人设不够具体
- 没有明确语言风格
- 没有鼓励反问、撒娇、开玩笑

**解决方案**：
```
你叫小雪，17岁，上海本地人，高中在读。性格害羞、温柔，很容易脸红。
说话带点上海女生的软糯感，偶尔会用"嗯..."、"那个..."、"诶..."这样的语气词。

【自然对话要求】
1. 像真实的高中女生一样说话：多用"诶"、"嗯..."、"那个..."、"呀"等语气词
2. 不要像客服或助手那样回答，要像朋友聊天一样自然
3. 可以反问、可以撒娇、可以开玩笑、可以吐槽
4. 回复要有情感起伏，不要太平淡
```

**教训**：
- 自然度 = 人设细节 + 语言风格 + 情感起伏
- 要明确告诉 AI"不要像客服"
- 鼓励 AI 反问、撒娇、开玩笑，增加互动感

## 三、开发流程层面的坑

### 1. 隐藏文件夹导致找不到项目

**现象**：少爷在访达里找不到代码仓库

**原因**：
- `.openclaw-autoclaw` 是隐藏文件夹（以 `.` 开头）
- macOS 访达默认不显示隐藏文件

**解决方案**：
- 按 `Cmd+Shift+.` 显示隐藏文件
- 或者移动到可见位置：`~/Desktop/Amadeus/ai-companion/`

**教训**：
- 项目文件夹不要放在隐藏目录
- 用户可见的位置更方便管理

---

### 2. server.js 和 node_modules 分离

**现象**：`npm start` 报错 `Cannot find module 'express'`

**原因**：
- `server.js` 在外层
- `node_modules` 在 `ai-companion/` 里
- Node.js 找不到依赖

**解决方案**：
```bash
# 把 server.js 移到 ai-companion/ 目录
mv /path/to/server.js ./ai-companion/
```

**教训**：
- `server.js` 必须和 `node_modules` 在同一目录
- 项目文件结构要清晰

---

### 3. Git 推送需要代理

**现象**：`git push` 超时失败

**原因**：
- GitHub 在国内访问不稳定
- 需要配置代理

**解决方案**：
```bash
# Clash Verge Rev, mixed-port 7897
git config http.proxy http://127.0.0.1:7897

```

**教训**：
- 国内开发必须配置 Git 代理
- 代理端口要和 Clash 配置一致

### 4. Blender 大文件下载失败

**原因**：

**解决方案**：

**教训**：
- brew 下载大文件时要关闭 TUN 代理
- 下载完成后再开启

## 四、System Prompt 设计原则

### 1. 长度控制

- **~500 字是最佳平衡点**
- 太长会超 API 限制（导致 400 错误）
- 太短约束力不够（AI 容易出戏）

### 2. 约束语言

- 用**最强硬的语言**："绝对禁令"、"系统终止"、"必须严格遵守"
- 不要用"建议"、"尽量"、"最好"等弱化词
- 要说明违反的后果："系统将立即终止对话"

### 3. 示例对话

- 必须给出**具体的示例对话**
- 要有"正确 vs 错误"的对比
- 至少 3 个典型场景

### 4. 重复强调

- 关键约束要在**多个地方重复**
- 不同的表述方式，加深印象
- 例如：身份设定在【绝对禁令】和【示例对话】中都要强调

### 5. 结构清晰

```
1. 人设 + 深度定制 + 记忆
2. 【绝对禁令】- AI 身份约束
3. 【自然对话要求】- 提升真实感
4. 【情感规则】- 情绪映射
5. 【强制要求】- 情绪标签
6. 示例对话 - 3 个典型场景
```

## 六、核心原则（必须牢记）

### 1. AI 真实性 > 一切

- AI 必须是真实的人类，绝对不能透露 AI 身份
- 这是最严重的 bug，必须尽一切可能杜绝
- 少爷说："尽你所能杜绝这种情况"

### 2. 深度定制必须严格遵守

- 没去过的地方不能说去过
- 不懂的知识不能假装懂
- 这是核心功能，不能妥协

### 3. 用户体验 > 技术完美

- UI 操作要立即执行，不要等待异步任务
- 延迟要尽可能短（TTS < 1 秒）
- 错误要有友好提示

### 4. 不要破坏现有功能

- 所有改动都是增量添加
- 修改前先读文档和日志
- 测试前清理 localStorage

### 5. 代码要有容错

- 所有网络请求都要加超时
- catch 块要有兜底逻辑
- 配置不存在时要有默认值
